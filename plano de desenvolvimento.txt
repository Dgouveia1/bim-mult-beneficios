Plano de Desenvolvimento - Sistema Bim Benefícios

Este documento detalha o plano para implementar as atualizações solicitadas, visando melhorar a performance, usabilidade e auditoria do sistema.

Fase 1: Otimização de Performance e Usabilidade (Busca de Clientes)

Objetivo: Alterar a lógica de carregamento de clientes para um modelo sob demanda, melhorando drasticamente a performance inicial da página de Clientes.

Tarefa: Mudar a lógica de carregamento de clientes para busca a partir de 3 dígitos.

Análise do Cenário Atual:

Atualmente, o arquivo js/clientes.js na função loadClientsData() carrega todos os clientes e seus dependentes para a memória (variável allPeople).

A busca é feita no lado do cliente (client-side) pela função filterAndRenderClients(), o que se torna lento com o aumento da base de dados.

Plano de Ação:

Backend (Supabase):

Garantir que as colunas nome, sobrenome, cpf e telefone na tabela clients possuam índices. Isso é crucial para a velocidade da busca no banco de dados.

Frontend (js/clientes.js):

Modificar a função loadClientsData() para que, por padrão, ela carregue apenas os 25 clientes mais recentes, em vez de todos. Isso garante que a tela nunca fique vazia ao carregar.

Alterar o event listener do campo clientsSearchInput em js/main.js.

A lógica atual chama filterAndRenderClients(). A nova lógica fará o seguinte:

Se o texto na busca tiver 3 ou mais caracteres, ele chamará uma nova função, como searchClientsOnServer(searchTerm).

Se o texto tiver menos de 3 caracteres, ele limpará os resultados ou voltará a exibir a lista inicial (clientes mais recentes).

Criar a função searchClientsOnServer(searchTerm) em js/clientes.js. Esta função irá:

Executar uma query no Supabase usando .or() para buscar o searchTerm nas colunas nome.ilike.%${searchTerm}%, cpf.ilike.%${searchTerm}%, etc.

Atualizar a tabela diretamente com os resultados retornados pelo banco de dados, em vez de filtrar o array allPeople.

Benefícios:

Carregamento Inicial Rápido: A página de clientes será carregada quase instantaneamente.

Escalabilidade: O sistema suportará dezenas de milhares de clientes sem perda de performance.

Busca Eficiente: A busca será mais rápida, pois é executada pelo banco de dados otimizado.

Fase 3: Modernização (Agenda em Tempo Real)

Objetivo: Tornar a agenda mais dinâmica e responsiva a alterações de múltiplos usuários simultaneamente.

Tarefa: Atualizar as funções de agenda para serem em tempo real.

Análise do Cenário Atual:

O arquivo js/agenda.js já utiliza o sistema de subscriptions do Supabase. Ele está configurado para recarregar todos os agendamentos (loadAppointments()) sempre que uma mudança ocorre. Isso funciona, mas pode causar um "piscar" na tela.

Plano de Otimização:

Modificar o callback da inscrição do Supabase em loadScheduleView() para ser mais inteligente.

O payload recebido do Supabase contém eventType ('INSERT', 'UPDATE', 'DELETE') e os dados (new, old).

Para INSERT: Em vez de redesenhar tudo, apenas crie o card para o agendamento em payload.new e adicione-o à coluna correta.

Para UPDATE: Encontre o card existente (document.querySelector('[data-appointment-id="${payload.new.id}"]')) e atualize suas informações (nome, horário, cor) com base nos dados de payload.new.

Para DELETE: Encontre o card correspondente ao payload.old.id e simplesmente o remova do DOM.

Benefícios:

Experiência Fluida: As atualizações na agenda aparecerão suavemente, sem recarregar a tela inteira.

Performance: Reduz o processamento no navegador e o tráfego de dados, pois manipula apenas os elementos que mudaram.

Fase 4: Qualidade de Vida e UX (Impressão de Exames)

Objetivo: Garantir que os pedidos de exame sejam sempre legíveis e bem formatados, independentemente da quantidade de itens.

Tarefa: Ajustar o tamanho da fonte ou layout dos pedidos de exame.

Análise: A geração de PDF ocorre em triggerPrintFromElement no arquivo js/pacientes.js usando a biblioteca jsPDF. A lógica atual não trata o excesso de conteúdo.

Plano de Ação (Implementar uma das duas opções ou ambas):

Opção A: Fonte Dinâmica

Antes de renderizar o texto dos exames no PDF, conte quantos exames existem.

Crie uma lógica condicional: se examList.length > 20, por exemplo, reduza o tamanho da fonte com pdf.setFontSize(9). Se for maior que 30, pdf.setFontSize(8), e assim por diante.

Calcule a altura total que o texto ocupará e, se exceder o limite da página, reduza a fonte.

Opção B: Layout em Duas Colunas (Recomendado)

Se a lista de exames (examList) ultrapassar um número definido (ex: 25 itens), divida a lista em duas.

Renderize a primeira metade da lista na posição x = 20.

Depois, mude a posição do "cursor" do PDF para a segunda coluna (x = 110) e renderize a segunda metade da lista.

Benefícios:

Profissionalismo: Evita documentos cortados ou com texto sobreposto.

Legibilidade: Garante que todos os exames solicitados estejam visíveis no documento impresso.

Fase 5: Melhorias Gerais Sugeridas

Objetivo: Abordar outros pontos de melhoria no sistema para aumentar a robustez e a qualidade.

Notificações "Toast":

Problema: O uso de alert() para feedback (ex: "Cliente salvo com sucesso!") interrompe o fluxo do usuário.

Sugestão: Implementar um sistema de notificações "toast" (pequenas mensagens que aparecem no canto da tela e somem sozinhas). Existem bibliotecas leves para isso, ou pode ser criado com CSS e JS puros.

Refatoração do Código de PDF:

Problema: A lógica para criar o layout do PDF (cabeçalho, rodapé, dados do paciente) está repetida para cada tipo de documento em js/pacientes.js.

Sugestão: Criar uma função auxiliar, como createPdfLayout(pdf, title, patientData, professionalData), para centralizar a criação do layout base do documento, reduzindo a duplicação de código.

Segurança (RLS - Row Level Security):

Problema: A política de segurança para a tabela appointments permite que qualquer usuário autenticado leia todos os agendamentos.

Sugestão: Refinar as políticas de RLS no Supabase. Por exemplo, um usuário com a role medicos só deveria poder ler (SELECT) os agendamentos onde o professional_id corresponde ao seu próprio ID de profissional.
